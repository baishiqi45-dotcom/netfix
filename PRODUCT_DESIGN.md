# netfix 产品设计文档（Phase A）

> 目标：把 netfix 从技术可用的原型，升级为「非技术用户也能独立使用」的 macOS 网络自救产品。

---

## 1. 北极星指标与成功标准

### 北极星指标

**首次用户从打开 App 到完成一次成功诊断的时间 ≤ 30 秒。**

这个指标直接对应用户的核心动机：网坏了，快点知道怎么了。

### 辅助指标

| 指标 | 目标值 | 说明 |
|---|---|---|
| 首次打开到出现主界面 | ≤ 3 秒 | 不能让用户怀疑 App 没反应 |
| 一键诊断完成率 | > 80% | 用户点了「诊断」后能跑完 |
| 诊断结果用户能复述原因 | > 70% | 用大白话，不是技术dump |
| 一键修复成功率 | > 50% | 至少解决最常见的 DNS/代理小问题 |
| 7 日留存 | > 30% | 用户愿意留在菜单栏 |

## 2. 目标用户与人设

### 2.1 主要人设：「小白 AI 用户」小 A

- 年龄：25-35
- 职业：设计师 / 自媒体 / 初级程序员 / 产品经理
- 设备：MacBook Pro / Air，macOS 13+
- 日常：每天开 ChatGPT / Claude / Cursor / YouTube
- 网络：买了机场/节点，但不懂得怎么配置
- 痛点：经常某个 AI 网站打不开，不知道是自己电脑问题还是节点问题
- 诉求：点一下就知道问题在哪，能自动修最好

**典型场景：**

> 小 A 正在用 Cursor 写代码，突然 Copilot 不响应了。他打开 netfix，看到菜单栏图标变黄，点一下，弹出「Claude API 连不上，可能是当前节点被风控」，然后点「一键修复」，自动切换到延迟最低的节点，30 秒后恢复。

### 2.2 次要人设：「效率极客」老 K

- 职业：全栈工程师 / 架构师
- 日常：用 Claude Code / Codex CLI / Kimi Code
- 诉求：不想浪费时间手动排查网络；希望 Agent 能调用 netfix 自动修

**典型场景：**

> 老 K 在 Kimi Code CLI 里问：「为什么我的 GitHub Copilot 今天抽风？」Kimi 调用 netfix_mcp，返回诊断结果，并自动执行修复。

### 2.3 企业人设：小团队 Leader

- 团队 5-20 人，远程/跨国协作
- 诉求：团队成员网络问题频发，希望有统一诊断工具；需要审计日志

## 3. 用户旅程地图

### 3.1 首次使用旅程

| 阶段 | 用户心理 | App 行为 | 设计目标 |
|---|---|---|---|
| 下载/打开 | 试试看 | 正常 App 启动，Dock 出现图标 | 建立「这是普通软件」的心智 |
| 欢迎页 | 这能干嘛？ | 一句话价值 + 大按钮「检查我的网络」 | 3 秒内明白价值 |
| 权限请求 | 会不会乱搞我电脑？ | 只请求当前步骤需要的权限，解释原因 | 建立信任 |
| 首次诊断 | 快点出结果 | 进度条 + 实时部分结果 + 预计时间 | 减少焦虑 |
| 结果页 | 到底什么问题？ | 一句话结论 + 四张状态卡片 | 让用户能复述问题 |
| 修复页 | 能不能一键修好？ | 安全修复自动执行，危险的让用户确认 | 证明价值 |
| 修复成功 | 好了吗？ | 重新检测，显示「已恢复」 | 闭环验证 |
| 日常 | 我还想放着 | 菜单栏常驻，状态灯一目了然 | 成为习惯 |

### 3.2 故障发生时的旅程

| 阶段 | 用户行为 | App 响应 |
|---|---|---|
| 发现故障 | ChatGPT 打不开 / Codex 报错 | 菜单栏图标变黄/红 |
| 寻求帮助 | 点菜单栏图标 | 弹出诊断面板，显示上次结果 |
| 诊断 | 点「一键诊断」 | 跑完整检测，10-20 秒 |
| 理解 | 看结果页 | 大白话根因 + 影响的 App |
| 修复 | 点「一键修复」 | 自动执行 Tier 1/2 修复 |
| 验证 | 等待 | 自动重测，显示修复结果 |
| 失败 | 修不好 | 给出精确手动步骤 + 复制报告 |

## 4. 交互设计原则

### 4.1 核心原则

1. **救援优先于监控**  
   用户打开 netfix 是因为出问题了，不是来看图表。默认界面必须是「诊断 + 修复」。

2. **一层 disclosed**  
   默认只显示：状态、结论、一个主要操作。技术细节全部折叠。

3. **状态即语言**  
   用颜色 + 图标 + 文字共同表达状态，不依赖单一视觉通道。

4. **操作有反馈**  
   每次点击都要让用户知道：已开始 / 进行中 / 已完成 / 失败。

5. **错误不甩锅**  
   永远说「发生了什么」「可以怎么做」，不说「你配置错了」。

### 4.2 文案风格

- 用主动句：「netfix 正在检查 DNS」
- 用用户能懂的话：「网站地址查不到」而不是「NXDOMAIN」
- 用具体数字：「DNS 响应 240ms，建议 < 50ms」
- 每个按钮文案都是动作：「检查网络」「立即修复」「查看步骤」「复制报告」

### 4.3 视觉设计系统

#### 颜色

| 含义 | SwiftUI | 使用场景 |
|---|---|---|
| 成功/正常 | `.green` | 状态灯、通过项 |
| 警告/降级 | `.orange` | 需要注意但未完全失败 |
| 失败/阻断 | `.red` | 连不上、修复失败 |
| 信息/中性 | `.blue` | 提示、链接、进行中 |
| 次要文字 | `.secondary` | 说明、时间戳 |

> 所有状态必须同时有图标和文字，不能只靠颜色。

#### 字体

- 标题：`.title2` / `.headline`
- 正文：`.body`
- 数据（IP、延迟）：`.body.monospacedDigit()`
- 辅助说明：`.caption`

#### 图标

- 使用 SF Symbols
- 状态图标：`checkmark.circle.fill`、`exclamationmark.triangle.fill`、`xmark.shield.fill`、`wifi.slash`
- 菜单栏图标：`network` 或 `antenna.radiowaves.left.and.right`

#### 布局

- 主窗口宽度：420-480pt
- 状态卡片：2×2 网格
- 主要操作按钮：底部固定工具栏
- 详细日志：可折叠面板

## 6. Onboarding 流程

### 6.1 首次启动 Onboarding

Step 1: 欢迎 + 价值
- 标题：让 Mac 网络问题一键可修
- 副标题：检查 Wi-Fi、DNS、代理，自动修复常见问题
- 按钮：检查我的网络 / 稍后设置

Step 2: 权限说明（仅当需要时）
- 如果用户点击「检查我的网络」，再请求必要权限
- 每个权限一页：
  - 图标 + 一句话解释
  - 「为什么需要」说明
  - 「继续」按钮（打开系统设置）
  - 「跳过」按钮（使用有限检测）

需要的权限：
- 本地网络访问（检测网关/DNS）
- 通知权限（可选，修复完成提示）
- 辅助功能/管理员权限（仅执行修复命令时）

Step 3: 首次基线检测
- 自动跑一遍完整诊断
- 让用户看到 App 能做什么
- 如果一切正常，显示「你的网络很健康」

### 6.2 非首次 Onboarding

- 如果用户跳过了首次检测，下次打开主界面时显示：
  - 提示卡片：「还没完成首次检测，点这里开始」
  - 不强制，不阻断

## 7. Agent（MCP）UX 设计

### 7.1 工具暴露原则

| 风险等级 | 工具示例 | 默认策略 |
|---|---|---|
| 只读 | ping、dns_lookup、check_service、get_report | Agent 可直接执行 |
| 低风险修改 | flush_dns、renew_dhcp、reset_proxy | 执行前展示计划 |
| 高风险修改 | switch_proxy_node、disable_ipv6、restart_interface | 必须用户确认 |
| 系统级 | install_helper、modify_system_proxy | 每次确认 |

### 7.2 Agent 调用时的用户可见流程

用户输入：「我 ChatGPT 打不开了」

Agent 响应：

```
我帮你检查一下。netfix 会执行：
1. 检测当前代理状态
2. 测试 ChatGPT 可达性
3. 检查 DNS 和 IPv6 泄漏

这些操作都是只读的，不会修改你的设置。
[开始检查]
```

执行中：

```
⏳ 正在检查…
✅ 代理核心运行正常
✅ DNS 解析正常
🟡 IPv6 可能绕过代理
❌ ChatGPT 返回 403
```

结果：

```
问题：ChatGPT 拒绝了你的请求，可能是因为当前 IP 被风控。
建议：切换到延迟最低的节点。
[切换到健康节点]  [查看详细报告]
```

如果涉及修改：

```
我需要帮你切换到另一个节点，这会修改 v2rayN 的配置并重启它。
[确认切换]  [取消]
```

### 7.3 信任设计

- 所有 Agent 操作本地记录，用户可在 App 内查看历史
- 修改类操作默认需要确认
- 提供「撤销」按钮或手动回滚步骤
- 不在云端上传任何网络数据

## 8. 技术执行架构

### 8.1 现有架构回顾

```
SwiftUI App
    ↓ NSTask / HTTP
Python Backend (netfix)
    ↓ subprocess / API
bin/*.sh + system commands
```

### 8.2 Phase A 需要新增/调整

| 模块 | 当前状态 | Phase A 动作 |
|---|---|---|
| SwiftUI App | 有基础窗口 + 菜单栏 | 重做 onboarding、主界面、状态卡片、设置页 |
| App 启动 | 有时需要 Control+右键 | 确保双击直接打开，Dock 可见 |
| 后端 API | HTTP + MCP 已有 | 优化 `/run` 同步调用体验，减少等待 |
| 诊断速度 | 10-15 秒 | 并行化、缓存上次结果、显示进度 |
| 中文报告 | 已有基础 i18n | 继续打磨文案，减少技术术语 |
| 服务列表 | 已有 rules/services.json | 增加「AI 服务急救包」预设 |
| 一键修复 | 有 fix_engine | 增加更多 Tier 1/2 修复项 |
| 更新机制 | 无 | Phase A 不做，后续加 Sparkle |

### 8.3 推荐技术栈

- **GUI**: SwiftUI + AppKit（NSStatusItem、NSPopover、NSWindow）
- **后端**: Python 3.9+ 标准库
- **通信**: 本地 HTTP（Agent/GUI 通用）
- **打包**: PyInstaller / 直接复制 Python 到 App Resources
- **签名**: Apple Developer ID + Notarization（未来）
- **更新**: Sparkle 2（未来）

### 8.4 文件组织

```
gui/macos/Netfix/
├── Sources/
│   ├── App/                 # App 生命周期、菜单栏
│   ├── Views/               # SwiftUI 视图
│   ├── Models/              # 数据模型
│   └── Services/            # 后端通信
├── Resources/
│   └── netfix_backend/      # Python 后端
└── Netfix.entitlements
```

## 9. 实现顺序与里程碑

### Sprint 1：启动体验（2 周）

目标：用户双击 App 能正常打开，看到主窗口。

- [ ] 修复 App 启动问题，确保 Dock 图标和窗口正常出现
- [ ] 设计并实现欢迎页
- [ ] 实现权限请求流程（本地网络、通知）
- [ ] 首次启动自动打开窗口
- [ ] 菜单栏图标常驻，点击打开窗口

验收：5 个真实用户能独立打开 App 并完成首次检测。

### Sprint 2：诊断体验（2 周）

目标：诊断流程清晰、快速、结果易懂。

- [ ] 主界面四张状态卡片
- [ ] 诊断进度条 + 实时步骤
- [ ] 结果页一句话结论
- [ ] 详细信息折叠面板
- [ ] 服务分组状态列表
- [ ] 诊断结果缓存/上次报告

验收：用户能在 30 秒内理解诊断结果。

### Sprint 3：修复体验（2 周）

目标：一键修复真正有用。

- [ ] 实现/接入 flush DNS、renew DHCP、toggle system proxy
- [ ] 增加「AI 服务急救包」规则（域名列表、QUIC 检测、IPv6）
- [ ] 修复前预览 + 修复后验证
- [ ] 修复失败时给出精确手动步骤
- [ ] 手动步骤支持复制

验收：常见 DNS/代理小问题能一键解决。

### Sprint 4：设置与打磨（2 周）

目标：产品完整、稳定、可发布内测。

- [ ] 设置页（通用、服务选择、修复选项、权限、关于）
- [ ] 启动时登录选项
- [ ] 通知权限与修复完成通知
- [ ] 全局快捷键
- [ ] 暗色模式适配
- [ ] 更新 README、build script、签名流程

验收：通过 10 人内测，主要问题修复。

## 10. 风险与应对

| 风险 | 应对 |
|---|---|
| App 启动/签名问题 | 先保证本地 build 可用；签名/公证放到内测后 |
| 诊断速度慢 | 并行化 + 进度条 + 缓存 |
| 用户不信任权限请求 | 只请求必要权限，详细解释，可跳过 |
| 修复成功率低 | 明确告知哪些能自动修、哪些只能手动 |
| 竞品模仿 | 靠社区 case、Agent 集成、macOS 原生体验建立壁垒 |
