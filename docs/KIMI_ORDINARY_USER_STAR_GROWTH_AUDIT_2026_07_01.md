# Netfix 普通用户 + 开源传播 + Star 增长体验审计

> 角色：最普通目标用户 + 中文产品体验杀手 + 开源传播审查员
> 时间：2026-07-01
> 项目路径：`<repo>`
> 审查方式：多视角走查 + 子智能体并行审查 README / Mac App / CLI / MCP / 代理解析 / 隐私安全

---

## 一、总体结论（先给答案）

Netfix 的技术闭环已经跑通：本地诊断、Keychain 存密码、脱敏报告、回滚、MCP 接入都有代码支撑，测试也过了。但 **GitHub 首屏和 App 首屏仍然在跟工程师说话**，普通用户 20 秒内会困惑三件事：

1. 这到底是卖代理的，还是帮我修网络的？
2. 我该下载 App、跑命令，还是接 MCP？
3. 它会不会把我电脑搞坏、把我密码偷走？

目前最大的传播阻力不是功能，而是 **首屏信息架构、人话文案和信任前置**。把这三件事改好，Star 和传播力会明显提升。

**一句话判断**：值得 Star，但首屏得重写；App 值得用，但保存/部署两步和错误引导得改。

## 二、README 中文首屏审查

### 2.1 第一眼能不能看懂？

**普通用户原话**：
- 「标题里又是 AI 又是 layer，我妈看了直接关掉。」
- 「macOS 上 Codex / ChatGPT / GitHub 突然连不上时……这句话我读完不知道跟我有什么关系。」
- 「我只有一个代理账号，想让我 Mac 能上网，README 却先说 Codex？」

**刺耳问题**：首屏价值主张把「AI 工具断线急救」放在前面，把「帮我配代理」放在后面。对最大量的普通用户（买了代理不会配）来说，顺序反了。

**修改建议**：
- 标题区改成「**买了代理不会配？粘贴一行，让 Mac 安全上网**」这类普通人能翻译成人话的 tagline。
- 副标题直接说三件事：粘贴代理参数 → 检测 → 部署到 Mac。
- 把 Codex/MCP/开发者接口放到第二屏以后。

### 2.2 是否过度技术化？

**普通用户原话**：
- 「‘OSI 五层’‘DNS、系统代理、代理核心、IPv6、TLS’——第一屏就给我上课？」
- 「‘本地网络急诊’听着像医院挂号，我就想让电脑上网。」
- 「‘MCP ready’ 是什么？徽章我不认识。」

**刺耳问题**：首屏同时出现 layer、DNS、IPv6、TLS、MCP 等术语，信息密度过高，普通人 20 秒消化不了。

**修改建议**：
- 第一屏只出现普通人听得懂的词：网络、代理、检测、修复、恢复。
- 技术词放到展开/折叠区或二级页面。
- badges 保留 `macOS`、`privacy: local-first`、`MIT` 即可，`MCP ready` 放到开发者小节。

### 2.3 第一屏知不知道痛点、结果、安装方式？

**普通用户原话**：
- 「第一屏我看到两个 `curl | bash`，一个给 Codex，一个给 App，我该复制哪个？」
- 「‘现在能怎么用’这一节太长了，我翻了两屏还没找到下载按钮。」
- 「那个 unsigned QA 版写在正文小字里，我根本不会细读。」

**刺耳问题**：安装入口不单一、不明确、不醒目；未签名 QA 版的提示不够警告级别。

**修改建议**：
- 第一屏只有一个主 CTA：`下载 Netfix.app`（或 `一键安装 macOS App`）。
- 把 Codex/MCP 安装命令折叠到「开发者 / Agent 用户点这里」。
- 在 `curl | bash` 命令上方加醒目 warning block：
  > ⚠️ 当前是 v0.2.0-qa.1 预览版，DMG 尚未完成 Apple Developer ID 签名与公证。首次打开需在「系统设置 → 隐私与安全性」点击「仍要打开」。

### 2.4 会不会被误解成卖代理 / 翻墙 / 绕风控？

**普通用户原话**：
- 「看到‘买到或自建了代理服务’，我第一反应这是机场客户端。」
- 「‘住宅 IP’‘切换到住宅 IP 节点’这种词，是不是在教人规避平台规则？」（用户原话摘录，仅作反例）
- 「GitHub topics 里有 clash、mihomo、sing-box，我 20 秒就归类成代理工具了。」

**刺耳问题**：README 和 explain.py 中多处措辞（买到代理、住宅 IP、切换到住宅 IP 节点）容易引发合规误会；GitHub topics 强化了这种印象。

**修改建议**：
- README 首屏第一句加边界声明：「**Netfix 不是代理客户端、不卖节点、严格遵守第三方平台规则。**」
- 「它解决什么」第二条改为：「已有合法代理凭据，但不确定如何在 macOS 上安全预检和配置。」
- explain.py 中「住宅 IP」「数据中心 IP 容易被风控」改为「当前出口属于数据中心 ASN，部分服务可能要求额外验证。可联系网络管理员或供应商切换到非数据中心节点。」
- GitHub topics 弱化 `clash`/`mihomo`/`sing-box`，或加说明「仅读取这些客户端运行状态，不是替代品」。

## 三、Mac App 用户路径审查

### 3.1 下载 / 安装 / 首次打开

**普通用户原话**：
- 「双击 DMG 弹出来‘无法验证开发者’，我敢不敢点‘仍要打开’？」
- 「安装完 Dock 里找不到，是不是没装上？」
- 「首页中间六个方块，我现在该点哪个？」

**刺耳问题**：Gatekeeper 拦截没有提前预警；安装引导缺少「拖到 Applications」图示；首页信息扁平，没有明确第一步。

**修改建议**：
- README/App 内增加 Gatekeeper 图文引导（系统设置 → 隐私与安全性 → 仍要打开）。
- DMG 背景加 Applications 替身图示。
- 首页改成「二选一」主入口：「我有代理账号」/「帮我检查网络」。

### 3.2 粘贴代理参数

**普通用户原话**：
- 「代理服务商后台给了 URL：`http://user:pass@host:port`，粘贴进去行不行？」
- 「‘参数类型’里的‘auto（常见 HTTP）’是什么意思？我不知道我的是 HTTP 还是 SOCKS5。」
- 「提示说‘不要只复制出口 IP’，那我到底该复制哪一行？」

**刺耳问题**：输入框没有实时解析反馈；参数类型 Picker 术语抽象；用户不知道从服务商后台复制哪一项。

**修改建议**：
- 粘贴后自动触发解析，显示「看起来是 HTTP 代理」「似乎少了密码」等即时提示。
- Picker 文案改为：「自动判断（大多数选这个）」「HTTP 代理」「SOCKS5 代理（服务商明确写 SOCKS5 时选）」。
- 输入框上方增加图文步骤：登录服务商后台 → 找到代理列表 → 复制包含地址/端口/用户名/密码的整行。

### 3.3 预检 / 保存 / 部署

**普通用户原话**：
- 「预检结果说‘1 条可用，0 条需修正’，那我现在能上网了吗？」
- 「点完‘保存到这台 Mac’，状态显示‘已保存……点下面部署到这台 Mac 开始用它上网’——所以保存不等于能用？」
- 「‘部署到这台 Mac’按钮旁边为什么有个黄色三角形？是不是危险？」
- 「保存和部署为什么要分两步？不能直接一步搞定吗？」

**刺耳问题**：保存和部署两步对用户不可见、不可理解；部署按钮带黄色警告三角，制造恐惧；预检结果用人话翻译不足。

**修改建议**：
- 顶部加可视化步骤条：`粘贴 → 检查 → 保存 → 开始使用`。
- 把「保存」和「部署」合并为一个用户操作：按钮文案「**开始使用这台 Mac 上网**」，内部先 save 再 apply。
- 部署按钮去掉黄色三角，改为蓝色主按钮。
- 预检结果直接说：「这个代理能连上，延迟 120ms」或「连不上，可能是密码错了」。

### 3.4 诊断 / 修复 / 回滚

**普通用户原话**：
- 「我点‘一键诊断’，它转了很久，我不知道它在干什么。」
- 「诊断完出现‘proxy_core_status’，这是啥？」
- 「‘处理建议’按钮是灰的，什么时候才能点？」
- 「回滚在哪？我在首页找了半天。」
- 「退出时突然弹窗‘这台 Mac 正在使用 Netfix 部署的代理’，我该点恢复还是仍然退出？」

**刺耳问题**：诊断中状态不透明；诊断 ID 可能 fallback 到英文内部名；修复入口状态不清晰；回滚入口隐蔽；退出弹窗三按钮吓人。

**修改建议**：
- 诊断进行中显示真实步骤文案，完成后 headline 直接写「网络本身正常，但 ChatGPT/GitHub 连不上」这种人话结论。
- 所有诊断 ID 必须有中文 displayName；原始 ID 只放在「查看技术详情」里。
- 底部「撤销」按钮改名「**恢复原来的网络设置**」，并在代理已部署时显示为醒目次级按钮。
- 退出弹窗简化为：「你现在正在用 Netfix 代理上网。退出后可能上不了网，要先恢复原来的网络设置吗？」按钮：「先恢复再退出 / 继续退出 / 取消」。

### 3.5 设置页 Tab 过多

**普通用户原话**：
- 「设置里 6 个 Tab，我找不到换 Key 的地方。」

**修改建议**：把 Settings 6 Tab 合并为 3 Tab：常规 / AI / 高级。

## 四、代理部署体验审查

### 4.1 用户到底该复制什么、不该复制什么？

**普通用户原话**：
- 「我不知道代理参数从哪里复制。」
- 「我粘贴了 Clash 订阅链接，提示 unsupported protocol: vmess，什么意思？」
- 「不要只复制出口 IP 这一点写了，但没告诉我不要复制 ss:// / vmess:// 链接。」

**刺耳问题**：README 和 App 内没有提前告诉用户支持哪些格式、不支持哪些格式；解析错误仍是英文工程文案。

**修改建议**：
- 输入框下方增加格式支持气泡：
  - ✅ 支持：`http://user:pass@host:port`、`socks5://...`、`host:port:user:pass`、`host,port,user,pass`
  - ❌ 不支持：`ss://`、`vmess://`、Clash/Mihomo/sing-box 订阅链接
- 检测到不支持的 scheme 时提示：「你粘贴的是 Clash/vmess 链接，Netfix 目前只支持 HTTP/SOCKS5 参数。请联系服务商索取 host:port:user:pass 四段参数。」
- 后端解析错误全部本地化，并附带下一步动作按钮（重新粘贴 / 查看示例 / 复制给客服）。

### 4.2 能不能让用户明确知道「这个代理能不能用、能不能部署到这台 Mac」？

**普通用户原话**：
- 「预检说 X 条可用，Y 条需修正，那我现在能上网了吗？」
- 「保存后只说‘可以部署到这台 Mac’，但没告诉我点哪里。」
- 「为什么保存了还不能上网？」

**修改建议**：
- 预检结果直接翻译成人话：「这个代理能用」「这个代理连不上，密码可能错了」。
- 保存后状态区分两步：
  - ✅ 已保存并测试通过（暂不改网络）
  - ⏳ 下一步：点「开始使用这台 Mac 上网」才生效
- 首页代理状态卡片增加「未部署 / 已部署」徽章。

### 4.3 失败信息有没有意义？

**普通用户原话**：
- 「密码错、端口错时，失败提示是英文，我不知道怎么办。」
- 「‘bridge_required’ 是什么？loopback 是什么？」

**修改建议**：

| 后端错误 | 当前文案 | 建议文案 |
|---|---|---|
| unsupported protocol: vmess | 英文 | Netfix 暂不支持 vmess 链接，请向服务商索取 HTTP/SOCKS5 参数 |
| port must be between 1 and 65535 | 英文 | 端口必须是 1-65535 之间的数字，请检查是否复制了多余字符 |
| host is required | 英文 | 缺少代理地址，请确认复制了完整一行 |
| username is required when password is provided | 英文 | 有密码但缺少用户名，请检查参数顺序 |
| proxy_auth_failed | 已部分中文化 | 账号或密码不对，建议重新复制服务商后台的最新密码 |
| dns_failed | 已部分中文化 | 代理地址解析失败，可能是地址写错了或本地 DNS 异常 |

每个错误卡片增加对应动作按钮。

## 五、AI 交互体验审查

### 5.1 不接 API Key 能不能用？

**普通用户原话**：
- 「我不知道 API Key 填在哪。」
- 「不接 API Key 是不是残血版？」
- 「AI 弹窗里‘我确认发送脱敏诊断报告’是什么？不勾就不能问？」

**刺耳问题**：用户不知道不接 Key 也能完整诊断；AI 上传 consent 文案抽象；无 Key 时的友好说明不够强。

**修改建议**：
- 首屏/首页明确提示：「**不接 API Key 也能完整诊断；Key 仅用于可选的云端 AI 解释。**」
- AI 弹窗无 Key 时显示：「你也可以不接 AI，Netfix 的诊断结果本身已经够用。」
- consent 文案改为：「允许 AI 查看当前网络检查结果（已隐藏密码等敏感信息）」。

### 5.2 接了 API Key 能做什么、不能做什么？

**普通用户原话**：
- 「接了 API Key，AI 能帮我什么？不能帮我什么？」
- 「我的 Key 会不会被发到你们服务器？」

**修改建议**：
- 在 AI 设置页加功能对照表：
  - 无 Key：本地诊断、本地规则解释、自动/手动修复、回滚
  - 有 Key：云端 AI 用人话解释脱敏报告、图片问诊
- 明确说明：Key 只存 macOS Keychain，Netfix 云端不存储、不中转。

### 5.3 AI 聊天框设计

**普通用户原话**：
- 「我就想问‘为什么 ChatGPT 打不开’，但输入框是空的，我不知道怎么问。」
- 「‘AI 服务急救包’和‘问 AI’有什么区别？」

**刺耳问题**：AI 入口有两个，容易混淆；输入框空着，没有预填问题；AI 回复可能只解释不指路。

**修改建议**：
- AI 问答页不应是开放式聊天，而应围绕「我的网络现在怎么了、下一步点什么」设计。
- 根据上下文动态预填问题：
  - 诊断上下文：「这句话是什么意思？」「是代理的问题吗？」「下一步我该点什么？」
  - 代理部署上下文：「这个代理能用吗？」「我粘贴的格式对吗？」
- AI 回复必须包含一句「下一步操作建议」，并映射到可点击按钮。

## 六、CLI / MCP / 安装体验审查

### 6.1 命令行子命令是不是人话？

**普通用户原话**：
- 「`triage` 是什么？急救分诊吗？」
- 「`doctor` 是给人看病还是给网络看病？」
- 「`kb` 是什么缩写？知识库？」
- 「‘IPv6 可能拖慢启动连接’‘根因’‘Tier 修复’这些词不是人话。」

**修改建议**：
- 给 `triage` 加别名 `check`，help 改成「快速排查最常见网络问题」。
- 给 `doctor` 加别名 `full-check`，help 改成「完整网络体检」。
- 给 `kb` 加别名 `guide` 或 `howto`。
- 给 `rollback` 加别名 `undo`。
- 报告文案：
  - 「根因」→「最可能的原因」
  - 「Tier 1/2/3」→「可自动执行 / 需要你确认 / 请手动操作」
  - 「置信度 0.85」→「很确定 / 比较确定 / 可能」
  - 「IPv6 可能拖慢启动连接」→「IPv6 没走代理，打开 App 可能变慢」

### 6.2 安装脚本敢不敢复制？

**普通用户原话**：
- 「我不知道这个命令会不会偷我密码。」
- 「它到底改了我电脑的哪些文件？」
- 「为什么默认拉的是 main 分支？明天代码变了怎么办？」

**修改建议**：
- 每个 installer 顶部加安全提示块：
  ```text
  安全提示：
    - 本脚本会：下载 Netfix / 写入 ~/.netfix / 注册本地 MCP 配置
    - 本脚本不会：读取或发送你的代理密码、API Key、浏览器数据
    - 可先运行：$0 --dry-run 预览所有操作
  ```
- 发布时附带脚本 SHA256，并提示用户先校验。
- `install_codex_mcp_from_github.sh` 默认 pin 到 release tag，不要默认 `main`。
- 三个脚本都支持 `--dry-run`。

### 6.3 MCP 注册到底做了什么？

**普通用户原话**：
- 「MCP 注册改了 Codex 的哪个文件？后悔了怎么卸载？」
- 「对 Codex 用户有什么好处？对 Kimi 用户有什么好处？」

**修改建议**：
- `install_mcp.sh` 注册后打印：Codex 配置文件位置 + 卸载命令 `codex mcp remove netfix`。
- 增加 `--remove` / `--uninstall` 参数。
- 对 Kimi 输出可直接复制的 JSON 配置块，而不是纯文字。
- `mcp_server.py` 工具描述中文化，并区分只读诊断 vs 会改系统设置的工具。

## 七、Star 增长审查

### 7.1 GitHub 上缺什么最影响 Star？

按影响程度排序：

1. **15 秒安装/使用 GIF**：首屏 hero 图是设计稿，不是真实 App 动起来的样子。用户想看「粘贴 → 检测 → 上网」全过程。
2. **中文首屏重写**：当前首屏像工程师文档，不像给普通人的产品页。
3. **安全/隐私速览前置**：20 秒用户需要立刻看到「不卖代理、密码存 Keychain、可回滚」。
4. **真实 case 展示**：`cases/` 目录已有案例，但 README 没利用。
5. **FAQ 小节**：回答「会不会改坏网络」「不接 Key 能不能用」「脱敏是什么意思」。
6. **对比表**：Netfix vs 手动配系统代理 vs Clash/V2RayN。
7. **MCP 示例一句话讲清好处**：不要只列命令，要说「注册后，你在 Codex 里问为什么连不上 GitHub，Codex 会自动调用 netfix」。
8. **英文 README 同步普通人化**：当前 repository description 仍是 engineer 腔。
9. **Release 页面明确 QA/unsigned 标注**：README 已写，但 release notes 也要重复。
10. **社交媒体传播图/长图**：适配小红书/即刻/Twitter 的中文用户路径长图。

### 7.2 项目标题、tagline、README 结构是否有传播力？

**普通用户原话**：

## 隐私与安全

- **本地优先**：诊断和规则推断默认在本地完成，不需要 LLM。

### 9.4 AI 问答页建议

**结论：AI 问答页不应是开放式聊天，而应围绕「我的网络现在怎么了、下一步点什么」设计。**

#### 建议结构

1. **上下文标题**
   - 来自诊断结果时：显示当前 headline，例如「检测到 2 个问题 · DNS 异常」
   - 来自代理部署时：显示「正在配置代理」

2. **预填问题（根据上下文动态变化）**

   诊断上下文：
   - 「这句话是什么意思？」
   - 「是代理的问题吗？」
   - 「下一步我该点什么？」

   代理部署上下文：
   - 「这个代理能用吗？」
   - 「我粘贴的格式对吗？」
   - 「为什么连不上？」

3. **输入框**
   - placeholder：`也可以直接描述你遇到的问题`
   - 不要留空就不能发送；留空时默认请求「解释当前结果」。

4. **上传报告 consent**
   - 文案改成：`允许 AI 查看当前网络检查结果（已隐藏密码等敏感信息）`
   - 如果未配置 AI Key，不强制勾选，而是提示「你也可以不接 AI，诊断结果本身已经够用。」

5. **结果展示**
   - AI 回复必须包含一句「下一步操作建议」，并尽可能映射到一个可点击按钮（例如「点‘处理建议’按钮」或「去‘粘贴代理’页面」）。
   - 不要让 AI 只给解释不给动作，否则用户看完还是不知道点哪里。
